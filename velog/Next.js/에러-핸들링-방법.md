## try-catch
나는 지금껏 try-catch 구문을 사용하여 에러 발생시 캐치해서 콘솔 에러로 출력하는 방식으로만 사용했다. 즉 에러 처리에 대한 뚜렷한 개념이 없었고 단순히 에러 발생 시 사용자가 에러를 직접 마주하지 않도록만 처리했었는데, 최근 원티드 프리온보딩 프론트엔드 역량향상 세션을 들으며 또 다른 방법을 알게 되었다.


## react-error-boundary 라이브러리
에러가 발생 시 특정 컴포넌트를 출력하는 방식이라고 한다.

[react-error-boundary 라이브러리 페이지 이동](https://www.npmjs.com/package/react-error-boundary)

위 페이지의 설명을 의역하면,
에러를 잡고(catch) 그에 대한 UI를 반환하고 싶은 컴포넌트들을 ErrorBoundary 컴포넌트로 감싼다. 주의할 점은, ErrorBoundary는 client 컴포넌트라서 "use client"를 선언한 페이지에서만 사용가능하다. 아니면 serializeable한 props로만 넘겨줄 수 있다고 하는데, 이 부분은 추가적인 이해가 필요하다.

> 이렇게 서버에서 fetching한 데이터는 클라이언트 컴포넌트에 props로 전달 가능합니다. 한 가지 유의해야 할 점은 **json으로 인코딩 가능한 serializable props만 전달 가능**하며 function은 전달할 수 없습니다.


즉, 클라이언트 페이지가 아닌 곳에서는 json으로 인코딩 가능한 serializable(직렬화) props만 전달 가능하다. _**serializable(직렬화)라는 것은 무엇일까?**_


> **객체를 파일에 저장하거나 파일에서 꺼내오기 위해서 또는 객체를 네트워크를 통해 전송하기 위해서는 미리 객체를 직렬화해야한다**


먼저 사용해보면서 차차 이해를 더해가야겠다.

### 설치방법
```
npm install react-error-boundary
```

### 가장 기본적인 사용 방법

```
"use client";

import { ErrorBoundary } from "react-error-boundary";

<ErrorBoundary fallback={<div>Something went wrong</div>}>
  <ExampleApplication />
</ErrorBoundary>
```

### 느낀점
아직 사용을 해본 적이 없어 상세하게는 모르겠다. 하지만 현재 진행중인 프로젝트에서 모든 에러를 try-catch 한 후, 모달을 띄워 새로고침을 유도하거나 메인페이지로 이동하는 식으로만 처리하는 나쁜 사용성을 제공하고 있기 때문에 적용하면서 포스트를 수정해나가겠다.(2024.03.24)


컴포넌트를 구현하여 fallback에 넣어주면 항상 그 컴포넌트가 드러나고, 이 때 다시 시도 또는 홈으로 이동 버튼 등의 방법으로 오류를 픽스할 수 있다. 위에서는 감만 잡았는데 실제로 사용했을 때도 동일한 사용법이었다. 추가적으로 이건 React에 지원되는 기능이라 Next.js 만 해왔던 내게는 처음 경험했을 수 밖에 없었고, 편리함에 압도당했다....신세계였다....(2024.07.08)
