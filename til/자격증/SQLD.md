# 1. 엔터티(ENTITY)


    > ### 데이터베이스?

    1. 여러 데이터들을 모아 통합적으로 관리하는 기술
    2. 여러 사람들이 함께 사용하고 공유 가능

    > ### 데이터 모델?

    - 현실세계의 대상을 추상화, 단순화, 명확화하여 데이터베이스로 표현하는 것

    ### 요괴의 눈물,,,

    1. 요구사항 접수
    2. 개념적 데이터 모델링
        1. Entity(개체) - Relationship(관계) - Attribute(속성)
    3. 논리적 데이터 모델링
    4. 물리적 데이터 모델링
    5. 데이터베이스에 저장할 수 있게 세팅 끝

    ---


    *(이미지 생략)*


    > ## 인스턴스(INSTANCE)?

    - 엔터티 집합 내에 존재하는 개별적인 대상

    > ## 엔터티(ENTITY, 개체)?

    - 업무에 필요한 정보를 저장/관리하기 위한 **집합적**인 명사 개념
    - 단순화 / 추상화 / 명확화

    ## 특징

    1. 반드시 업무에서 필요한 대상이고 업무에 사용될 것
    2. 유일한 식별자로 식별이 가능할 것
    3. 인스턴스가 2개 이상일 것
    4. 속성이 반드시 2개 이상 존재할 것
    5. 관계가 하나 이상 존재할 것 (단 코드성, 통계성은 생략 가능)

    ## 분류


    ### 유무형에 따른 분류

    1. 유형
        1. 물리적 형태가 있는 엔터티
    2. 개념
        1. 물리적 형태가 없는 엔터티
    3. 사건
        1. 업무 수행 중에 발생하는 엔터티 (제일 많이 발생)

    ### 발생시점에 따른 분류

    1. 기본/키
        1. 본래 업무에 존재하는 정보
        2. 독립 생성 가능, 주식별자 보유
        3. 직원, 고객, 상품
    2. 중심
        1. 기본 엔터티로부터 발생
        2. 업무에 있어 중심 역할
        3. 주문, 매출, 계약
    3. 행위
        1. 2개 이상 엔터티로부터 발생
        2. 주문이력

    ## 명명(naming) 규칙

    1. 가능한 현업 용어를 쓴다
    2. 가능하면 약어를 사용하지 않는다
    3. 단수 명사를 사용한다
    4. 엔터티 이름은 유일해야한다
    5. 엔터티 생성 의미대로 이름을 부여한다

# 2. 속성(ATTRIBUTE)


    > ### 속성(ATTRIBUTE)?

    - 업무상 관리하기 위해 의미적으로 더는 분리되지 않는 최소의 데이터 단위
    - 엔터티가 가지는 공통적인 특징을 표현
    - 우리가 관리하고자 하는 정보가 됨

    *(이미지 생략)*


    > ### 엔터티, 인스턴스, 속성의 관계

    1. 하나의 엔터티는 2개 이상의 인스턴스를 가진다. - 행 == 인스턴스
    2. 하나의 엔터티는 2개 이상의 속성을 가진다. - 컬럼명 == 속성
        1. 1개만 가질 순 있지만 db 사용 이유가 업슴
    3. 속성은 각 인스턴스를 설명해줄 수 있다.
    4. 하나의 속성에는 하나의 속성값만 들어간다.

    *(이미지 생략)*


    > ### 식별자?

    - 엔터티 내 유일한 인스턴스를 식별할 수 있는 속성의 집합

    *(이미지 생략)*


    > ### 속성의 분류 (특성에 따른 분류)

    1. 기본
        1. 업무로부터 추출한 속성으로 제일 많이 발생
    2. **설계(design)**
        1. 설계시 규칙화 등이 필요해 만든 속성
        2. 코드성이나 일련번호 등
    3. 파생
        1. 다른 속성들로부터 계산/변형되어 만들어진 속성

    > ### 속성의 분류 (구성방식에 따른 분류)

    - PK, FK, 일반속성, 복합속성

    > ### 속성 명명(Naming) 규칙

    1. 가능한 현업 용어를 쓴다.
    2. 가능하면 약어를 사용하지 않는다.
    3. 명사형을 쓰고 서술식이나 수식어 등을 제한한다.
        1. 오늘배송된상품 → 일배송상품
    4. 가능한 속성 이름은 전체 데이터 모델에서 유일해야한다.

    > ### 도메인(Domain)?

    - 각 속성이 입력받을 수 있는 값의 정의 및 범위
    - 보통 테이블을 만들 때 각 속성마다 자료형 및 제약조건을 줄 때 결정된다.
        - 나이는 숫자만 입력받을 수 있다. 입력값은 0~999까지로 한다.
        - 이름은 문자형으로 입력받을 수 있다. 최대 5자리만 받는걸로 한다.

# 3. 관계(Relationship)


    > ### 관계(Relationship)?

    - 엔터티 내의 인스턴스들 간에 서로 논리적인 연관성이 있는 상태

    > ### 관계의 분류 - 존재/행위


    *(이미지 생략)*

    - **ERD**는 존재/행위 등 관계를 위처럼 똑같이 표시
        - E(엔터티) - R(관계) - D(그림)
    - **UML 클래스 다이어그램**은
    연관관계(존재)는 실선으로 표기하고,
    의존관계(행위)는 점선으로 표시

    > ### 관계를 표시하는 방법

    - 표기하는 방식에 따라 **IE**와 **Barker** 표기법으로 구분됨
    - **관계명**(Membership)
    - **관계차수**(Cardinality/Degree)
    - **관계선택사양**(Optionality)
    1. 관계명을 표시한다. 이때 애매한 동사나 과거형은 피한다.
    2. 관계차수(1:1, 1:M, M:N)는 엔터티 내 각 인스턴스들이 얼마나 참여하는지를 의미한다.

        |—|, >--, >|—, 
        **N:M 관계 → 1:M, M:1로 엔터티를 쪼개야 함**

    3. 관계선택사양은 엔터티 내 각 인스턴스들이 필수/선택 참여하는지를 의미한다.
    >|0—| 여러직원일수도, 한명일수도, 없을수도 o — 반드시들어가야한다

# 4. 식별자 & 데이터 모델


    요괴의눈물


    요구사항 분석 - 개념적 데이터 모델링 - 논리적 데이터 모델링 - 물리적 데이터 모델링


    > ### 데이터 모델링이란?

    - 현실세계의 복잡한 대상을 단순하게 표현하기 위해 추상화, 단순화, 명확화 하여 일정표기법으로 표시하는 것
    - **계획/분석/설계** 시 업무분석 및 설계, 설명에 사용, **구축/운영** 시에는 변경, 관리 목적으로 사용

    ## 데이터 모델링 3단계 (개념 - 논리 - 물리)

    1. **개념적 데이터 모델링**
        1. 추상적
        2. 업무중심적
        3. 포괄적
        4. 전사적(EA)
        5. DBMS 독립적
        6. ERD 생성
    2. **논리적 데이터 모델링**
        1. KEY, 속성, 관계 등을 정의
        2. **재사용성이 높음**
        3. 특정 DBMS에 종속적
        4. 정규화 수행
    3. **물리적 데이터 모델링**
        1. 실제 DB에 이식되도록 성능, 저장 등 물리적 성격 고려
        2. 수준이 구체적임

    ### 개념적 모델링 방법


        > ### ERD  
        >   
        > 엔터티(Entity)와 엔터티 간의 관계(Relationship)를 발견하고  
        > 이를 그림(Diagram)으로 표현하는 행위로 **개념적 데이터 모델링의 결과물**.

        1. 엔터티를 그린다. 속성은 제외. 보통 중요한 엔터티를 왼쪽 상단에 둔다.
        2. 엔터티를 적절히 배치
        3. 관계를 설정한다. 서로 관계가 있는 엔터티끼리 선으로 이어준다.
        4. 관계명을 기술한다.
        5. 관계의 **참여도(Cardinality)**를 기술한다.

            참여도란 엔티티 안의 인스턴스들이 얼마나 참여하는지를 의미.
            **1:1, 1:N, N:N 관계** 등이 있으며, N쪽에는 까치발(**>**)로 표시한다.

        6. 관계의 필수여부(=Optionality)를 기술한다.
        필수 여부란 엔티티 안의 인스턴스들이 반드시 참여하는지를 의미.
        최소 참여는 (I)을 표시하고, 선택참여는 (O)으로 표시한다.

    > ### 식별자(Identifiers)란?

    - 엔터티 내 유일하게 인스턴스를 식별할 수 있는 속성의 집합

    ### 식별자 분류

    1. 대표성 여부
    주식별자 / 보조식별자
    2. 스스로생성(자생)여부
    내부식별자 / 외부식별자
                        (외래키)
    3. 단일속성여부
    단일식별자 / 복합식별자
    4. 대체여부
    본질식별자 / 인조식별자
    (부서명 / 부서코드)

    ### 주식별자 도출기준

    1. 업무에서 자주 쓰는 속성일 것 (≠ 주민번호) (= 직원 아이디)
    2. 명칭, 이름 등은 피할 것 → 중복을 피할 것
    3. 속성의 수가 많지 않을 것

    ### 주식별자 **특징**

    1. **유일성**: 인스턴스를 유일하게 식별하기
    2. **최소성**: 속성의 수가 많지 않기
    3. **불변성**: 가급적 변하지 않기
    4. **존재성**:  값이 꼭 입력되기

    > ### 데이터베이스 관련 용어 정리

    - **데이터베이스**(DataBase, DB)
        - 데이터를 일정한 형태로 저장해 놓은 것
    - **데이터베이스관리시스템**(DataBase Management System, DBMS)
        - 기존 데이터베이스 기능에 추가로 데이터 손상을 방지 및 복구, 인증된 사용자만 접근 등 추가 기능을 지원하는 관리 시스템
    - **관계형 DBMS** (Relational DBMS, RDBMS)
        - 테이블로 데이터를 관리하고 테이블간 관계를 이용해 데이터를 정의하는 방식으로 대부분의 기업이 사용하며 지금 공부하는 Oracle도 RDBMS 중 하나임
    - **테이블**(Table)
        - RDBMS에서 실제 데이터가 저장되고 조회되는 2차원 배열 형태의 저장소 공간
        - 엔터티, 속성, 인스턴스가 각각 DB가 이해할 수 있는 형태인 테이블, 컬럼, 튜플로 변경된 것
    - SQL (Structured Query Language)
        - RDBMS에서 데이터 정의, 조작, 조회, 제어 등을 하기 위해 사용하는 언어

        | **DDL** (Data Definition Language, 데이터 정의어)      | CREATE, ALTER, DROP, RENAME, TRUNCATE |
        | ------------------------------------------------ | ------------------------------------- |
        | **DML** (Data Manipulation Language, 데이터 조작어)    | SELECT, INSERT, UPDATE, DELETE, MERGE |
        | **TCL** (Transaction Control Language, 트랜잭션 제어어) | COMMIT, ROLLBACK, SAVEPOINT           |


    > ### 일반집합연산자와 순수관계연산자

            - —INTERSECTION으로 구현
        - DIFFERENCE

    - **순수관계연산자** 종류

        - (NATURAL) JOIN

        - DIVIDE

    > ### SELECT

    - 테이블에서 원하는 데이터를 조회할 때 사용하는 문법
    - SELECT ***** FROM TB_PRD;
        - TB_PRD 테이블의 **모든 컬럼** 정보를 출력합니다
    - SELECT **DISTINCT** PRD_TYPE FROM TB_PRD;
        - TB_PRD 테이블의 PRD_TYPE 컬럼을 기준으로 값을 중복없이 출력합니다.
    - **AS**
        - SELECT 부분에서 출력하려는 컬럼에 대해 **새로운 별명(ALIAS)를 부여**할 수 있습니다.
        - 사용 주의사항
            - 띄어쓰기 불가
            - 문자로 시작해야함
            - 예약어 불가
            - 특수문자는 $ , _ , # 만 가능
