**Domain + Infrastructure의 집합**


**FSD**


| **Layer**    | **역할**                    | **예시**                              |
| ------------ | ------------------------- | ----------------------------------- |
| **App**      | 전역 설정, 엔트리 포인트, 라우팅       | `app/index.ts`                      |
| **Pages**    | 페이지 컴포넌트, 레이아웃 조합         | `pages/home`                        |
| **Widgets**  | 복합적인 UI 구성 요소 (여러 기능 조합)  | `widgets/user-profile`              |
| **Features** | 사용자 상호작용 가능한 독립된 기능       | `features/add-to-cart`              |
| **Entities** | 비즈니스 도메인 모델 및 관련 로직       | `entities/user`, `entities/product` |
| **Shared**   | 전역적으로 재사용되는 유틸리티, UI 컴포넌트 | `shared/ui`, `shared/lib`           |


| **구분**               | **제시된 폴더 구조 (기술+도메인 혼합)**                                                                     | **FSD (Feature-Sliced Design) 아키텍처**                           |
| -------------------- | --------------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| **분리 기준**            | **기술적 관심사**가 메인 (`pages`, `components`, `hooks`, `api` 등) + **도메인** (`damain-name`)이 하위에 혼합됨. | **Layer(계층)**가 메인 + **Slice(도메인/기능)**가 Layer 내부에 수평적으로 존재.     |
| **의존성 규칙**           | **규칙이 명시적이지 않음**. (`components`의 컴포넌트가 `api`를 직접 참조하는 등)                                      | **단방향 의존성** 규칙이 매우 엄격함. (상위 Layer는 하위 Layer만 참조 가능)            |
| **최상위 폴더**           | 기술적 분류 (`pages`, `layout`, `components`, `api`, `store` 등)                                    | FSD Layer (`app`, `pages`, `features`, `entities`, `shared` 등) |
| **유지보수**             | 프로젝트가 커질수록 **기능(도메인) 간의 결합도**가 높아지기 쉬움.                                                       | **Slice(기능 단위)**가 독립적으로 관리되어 결합도가 낮고, 부작용 없이 수정 용이.            |
| **`shared`****의 위치** | `components`, `hooks`, `api`, `store` 등 **여러 곳**에 산재됨.                                        | **최상위 레이어** 중 하나로 명확히 분리되며, **순수한 코드**만 포함해야 함.                |


제시하신 구조는 **관심사별 분리(Separation of Concerns)** 원칙을 따르면서 **도메인(Domain)**과 **기술적 관심사(Technical Concerns)**를 혼합한 구조로, 흔히 **계층형(Layered)** 또는 **모듈형(Modular)** 구조로 불립니다.


이 구조는 **FSD (Feature-Sliced Design)**와 몇 가지 핵심적인 차이점을 가지며, 단순히 아토믹 디자인 패턴을 적용했는지 여부와는 별개의 문제입니다.

## 1. FSD와의 근본적인 차이점: 구조의 주체와 의존성 규칙


제시된 구조와 FSD의 가장 큰 차이점은 **무엇을 기준으로 코드를 분리하는지**와 **모듈 간의 의존성을 어떻게 제어하는지**에 있습니다.


### 1.1. 🧩 관심사 분리의 기준 (주체)


| **구분**     | **제시된 구조**                                     | **FSD (Feature-Sliced Design)**                          |
| ---------- | ---------------------------------------------- | -------------------------------------------------------- |
| **분리 기준**  | **기술적 역할** (Technical Role)                    | **영향력 범위와 추상화 수준** (Layers)                              |
| **최상위 폴더** | `pages`, `components`, `hooks`, `api`, `store` | `app`, `pages`, **`features`**, **`entities`**, `shared` |
| **핵심 문제**  | 기능(Feature)을 구성하는 코드가 여러 폴더에 **흩어져 있음**.       | 기능(Slice) 관련 코드가 한 곳에 **응집**되어 있음.                       |

- **제시된 구조:** 기능(예: 장바구니)을 완성하는 코드가 `components/domain-name`, `hooks/domain-name`, `api/domain-name` 등 **기술적 폴더에 분산**됩니다. 상품 아이템 컴포넌트(`components`)는 상품 리스트(`features`)가 사용하는 **종속적인 관계**입니다.
- **FSD:** 기능(Feature) 자체를 하나의 **레이어**(`features` 레이어)나 **슬라이스**(`features/cart`)로 묶고, 그 안에 필요한 UI/Hooks/API 세그먼트를 모두 포함합니다. **기능 중심의 응집도**를 높입니다.

### 1.2. 🔗 의존성 규칙 (단방향 vs. 자유로움)

- **제시된 구조:** 명확한 **의존성 규칙이 없습니다.** 예를 들어, `features`가 `hooks`를 사용하고 `hooks`가 `api`를 호출하는 것은 자연스럽지만, 이 구조 내에서 `components`가 `api`를 직접 호출하는 것을 **막는 규칙은 명시되어 있지 않습니다.**
- **FSD:** **단방향 의존성(One-way Dependency)**을 핵심 원칙으로 합니다. 상위 레이어는 하위 레이어만 참조할 수 있습니다. 이를 통해 **예상치 못한 부작용**을 방지하고 리팩토링 범위를 예측 가능하게 만듭니다.

## 2. 제시된 구조의 특징과 아토믹 디자인 패턴 적용


제시하신 구조는 **기술적 관점**에서 매우 잘 분리되어 있으며, 특히 `api`와 `store`를 도메인별로 나눈 점이 훌륭합니다.


### 2.1. ⚛️ 아토믹 디자인과의 관계


제시된 구조는 그 자체로 아토믹 디자인은 아니지만, **컴포넌트 폴더를 아토믹 디자인 패턴으로 구조화**할 수 있습니다.

- **제시된 구조:**
    - `components/shared`: **Atoms** (가장 작은 UI: 버튼, 인풋) + **Molecules** (작은 조합: 검색창)
    - `components/domain-name`: **Molecules** 또는 **Organisms** (도메인 특화된 UI: `ProductItem`)
- **`features`** **폴더의 역할:**
    - `features` 폴더는 아토믹 디자인의 **Templates**나 **Pages**에 해당하는 **복합적인 역할**을 수행합니다.
    - **Features = UI + 비즈니스 로직 조립:** 상품 리스트(`features/productList`)는 `components`의 `ProductItem`을 가져와 `hooks`에서 가져온 데이터를 반복문으로 돌리는 **조립(Composition) 역할**을 합니다.

### 2.2. FSD의 유사 요소


제시된 구조에서 `domain-name` 폴더들은 FSD의 **Slice** 개념과 유사하지만, FSD는 이 **Slice**를 **레이어(Layer)로 한 번 더 감싸서** 의존성을 엄격하게 제한합니다.


| **제시된 구조**          | **FSD Layer**            | **FSD Slice/Segment**      | **역할**          |
| ------------------- | ------------------------ | -------------------------- | --------------- |
| `pages/damain-name` | `pages`                  | `user-page/index.tsx`      | 페이지 수준 조립       |
| `features`          | `features`               | `add-to-cart/ui/`          | 사용자 인터랙션이 있는 기능 |
| `components/shared` | `shared`                 | `ui/button.tsx`            | 재사용 가능한 UI      |
| `api/domain-name`   | `entities` 또는 `features` | `user/api/` 또는 `cart/api/` | 데이터 및 서버 통신     |
| `hooks/domain-name` | `features` 또는 `entities` | `user/model/useUser.ts`    | 상태 및 비즈니스 로직    |


결론적으로, 제시된 구조는 **기술별 분리와 도메인 분리**를 혼합한 일반적인 대형 프로젝트 구조이며, FSD는 이 구조에 **Layer 개념과 단방향 의존성**이라는 강력한 규칙을 추가하여 **확장성과 응집도를 극대화**한 아키텍처 방법론이라고 볼 수 있습니다.
